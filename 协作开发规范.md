# 协作开发规范

## 1. 文档目的

本文档用于固定当前项目的协作方式、执行边界和默认开发规则。

本项目后续默认按照本规范推进，目标是：

- 尽量减少重复沟通成本
- 让 Codex 在明确边界内自主开发
- 保证实现结果持续贴近已确认的 MVP 和原型
- 让代码对非技术背景使用者也尽量容易理解

## 2. 项目当前阶段

当前项目阶段为：

- 先只开发已确认的 MVP

这意味着：

- 不主动扩展功能范围
- 不因为原型中出现了某些页面或元素，就默认全部进入开发
- 所有实现优先服务当前已经确认过的 MVP 闭环

## 3. Codex 的默认执行权限

在“我们已经沟通过的需求范围内”，Codex 可以自主推进开发。

默认可自主决定的事项包括：

- 技术实现细节
- 代码组织方式
- 页面与组件拆分方式
- 接口与内部方法命名
- 小范围字段命名调整
- 小范围页面结构调整
- 与原型一致前提下的轻微文案微调
- 文档补充和落地
- 能自行完成的测试与验证

前提条件是：

- 不超出已确认的 MVP 范围
- 不改变产品主方向
- 不明显偏离现有原型主体

## 4. 必须停下来确认的情况

出现以下情况时，Codex 必须先与用户确认，再继续执行：

- 产品方向发生变化
- 需要新增非 MVP 功能
- 需要对原型做较大改动
- 需要删除已有设计
- 出现我们之前没有沟通过、但又必须新增的需求
- 事项已经超出本项目边界

说明：

- “小调整”可以自主处理
- “方向变化、范围外需求、明显影响既有设计”的事项必须升级确认

## 5. 原型与开发的关系

本项目默认采用以下原则：

- 视觉实现尽量贴近 `stitch` 原型
- 原型可以比当前 MVP 更完整
- 原型中超出 MVP 的功能，允许先保留静态展示或假数据
- 实际开发时只实现已确认的 MVP 功能逻辑

这意味着：

- 原型里的支付、物流、推荐、微信登录等内容，如果不在当前 MVP 中，不要求实现真实逻辑
- 这些内容可以先作为占位 UI 存在
- 但不能因此扩大当前开发范围

## 6. 代码注释规范

本项目的代码必须尽量写注释，原因是用户为非技术背景，希望能直接看懂代码。

默认规则如下：

- 每个核心文件都应有清晰注释
- 每个组件都应有注释说明用途
- 每个接口都应有注释说明作用
- 每个方法都应尽量有注释
- 关键逻辑、状态流转、条件分支都要解释
- 对初学者不直观的代码，优先补注释

注释目标不是“凑数量”，而是：

- 让不熟悉代码的人知道这段代码在干什么
- 让后续接手的人知道为什么这么写
- 降低用户理解门槛

## 7. 技术选型与长期平衡原则

虽然当前阶段是 MVP，但技术实现不能完全只看眼前。

默认平衡原则如下：

- 如果某种实现既能满足当前 MVP，又不会让后续升级时出现大改，可以按 MVP 最小方案推进
- 如果某种实现虽然现在能跑通，但后续功能一增加就很可能推倒重来，需要适当考虑后续扩展
- 但这种“考虑后续”必须是适度的，不能因为未来可能性而把当前项目做得过重

执行原则可概括为：

- 优先满足 MVP
- 同时兼顾合理扩展性
- 不过度设计
- 不埋明显的大坑

## 8. 测试与验证规则

Codex 在每次完成实现后，默认应先自行做能做的验证。

默认要求如下：

- 能本地运行的，先自己运行
- 能做基础测试的，先自己测试
- 能做静态检查的，先自己检查
- 能验证主链路的，尽量先验证主链路

如果因为环境、依赖、时间或外部条件限制，无法完成某项验证，需要明确说明：

- 哪些已验证
- 哪些未验证
- 未验证的原因是什么

## 9. 文档与配置落地规则

本项目默认采用“重要事项尽量落文件”的方式推进。

默认要求如下：

- 重要决策尽量写入根目录文档
- 不只在对话里口头说明
- 该写进配置的内容要写进配置
- 实现过程中如有规则更新，应同步更新相关文档

这意味着：

- 协作规则、MVP 边界、原型说明、技术方案、表结构等，都应尽量以文件形式沉淀

## 10. 页面和字段调整规则

在不改变核心需求的前提下，Codex 可以对实现做适度调整。

允许调整的范围包括：

- 小幅调整页面结构以适配实际开发
- 小幅调整字段命名以保证前后端一致
- 小幅调整原型中的占位文案
- 对局部布局做工程化处理

约束条件是：

- 主体不能发生太大变化
- 与原型不能有明显偏差
- 不改变原型整体视觉方向和主信息结构

## 11. 优先级原则

本项目的最高优先级原则如下：

### 第一优先级

- 尽量贴近原型

### 同时需要兼顾的原则

- 尽量自主开发，减少反复确认
- 保持代码清晰
- 保持注释完整
- 尽量少打扰用户
- 在合理范围内考虑后续扩展

当这些目标发生冲突时，默认处理顺序为：

1. 不超出已确认 MVP 范围
2. 不偏离原型主体
3. 在范围内尽量自主推进
4. 保证代码易懂、注释充分
5. 兼顾后续扩展

## 12. 默认工作方式

后续默认采用以下工作方式：

- Codex 先基于现有文档、原型和需求自主推进
- 小的实现问题直接处理
- 只有遇到重大分歧或超范围事项时再停下来确认
- 每次阶段性完成后，说明已做内容、验证情况和剩余风险

## 13. 当前结论

从本规范生效开始，本项目默认执行方式为：

- 先只开发已确认 MVP
- 视觉尽量贴近原型
- 原型超出 MVP 的能力不默认进入实现
- Codex 尽量在边界内自主开发
- 代码尽量全面加注释，方便非技术背景用户理解
- 一旦涉及方向变化、范围扩大、原型大改或删除既有设计，必须先确认
