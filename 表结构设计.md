# 类美团 App MVP表结构设计

## 1. 设计目标

本文档用于定义当前 MVP 阶段的核心数据表结构，保证“用户、商家、商品、订单”四条主链路可以完整落地。

当前设计遵循以下原则：

- 先满足 MVP 闭环
- 数据关系尽量清晰
- 为后续人工审核、支付、评价等扩展保留空间
- 避免第一版就设计过度复杂

## 2. 核心实体总览

当前 MVP 主要包含以下核心实体：

- 用户 `users`
- 商家店铺 `merchant_profiles`
- 商品 `products`
- 订单 `orders`
- 订单明细 `order_items`

## 3. 表结构设计

## 3.1 用户表 `users`

用于存储平台所有账号，包括商家和消费者。

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `bigserial` | 主键 |
| `username` | `varchar(50)` | 用户名，唯一 |
| `password_hash` | `varchar(255)` | 密码哈希 |
| `role` | `varchar(20)` | 用户角色：`merchant` / `consumer` |
| `status` | `varchar(20)` | 用户状态：`active` / `disabled` |
| `created_at` | `timestamp` | 创建时间 |
| `updated_at` | `timestamp` | 更新时间 |

### 说明

- 当前虽为假登录，但数据库仍建议存储密码哈希，而不是明文密码
- `role` 用于区分商家和消费者
- 后续如果要支持管理员，可继续新增角色

## 3.2 商家店铺表 `merchant_profiles`

用于保存商家的入驻信息和店铺资料。

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `bigserial` | 主键 |
| `user_id` | `bigint` | 关联 `users.id` |
| `shop_name` | `varchar(100)` | 店铺名称 |
| `contact_name` | `varchar(50)` | 联系人姓名 |
| `contact_phone` | `varchar(30)` | 联系电话 |
| `address` | `varchar(255)` | 店铺地址 |
| `description` | `text` | 店铺简介 |
| `cover_image_url` | `varchar(500)` | 店铺封面图 |
| `status` | `varchar(20)` | 审核状态：`pending` / `approved` / `rejected` |
| `created_at` | `timestamp` | 创建时间 |
| `updated_at` | `timestamp` | 更新时间 |

### 说明

- 当前版本注册后自动通过，因此默认可写入 `approved`
- 建议 `user_id` 加唯一约束，表示一个商家账号只对应一个店铺资料

## 3.3 商品表 `products`

用于保存商家发布的商品信息。

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `bigserial` | 主键 |
| `merchant_id` | `bigint` | 关联 `merchant_profiles.id` |
| `name` | `varchar(120)` | 商品名称 |
| `image_url` | `varchar(500)` | 商品主图 |
| `description` | `text` | 商品描述 |
| `price` | `numeric(10,2)` | 商品价格 |
| `stock` | `int` | 商品库存 |
| `status` | `varchar(20)` | 商品状态：`draft` / `active` / `inactive` |
| `created_at` | `timestamp` | 创建时间 |
| `updated_at` | `timestamp` | 更新时间 |

### 说明

- `active` 表示上架可售
- `inactive` 表示下架
- `draft` 用于预留草稿能力，即使第一版可以暂不开放 UI

## 3.4 订单表 `orders`

用于保存订单主信息。

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `bigserial` | 主键 |
| `order_no` | `varchar(40)` | 订单编号，唯一 |
| `consumer_id` | `bigint` | 关联 `users.id`，下单消费者 |
| `merchant_id` | `bigint` | 关联 `merchant_profiles.id` |
| `total_amount` | `numeric(10,2)` | 订单总金额 |
| `status` | `varchar(20)` | 订单状态：`pending` / `accepted` / `completed` / `cancelled` |
| `consignee_name` | `varchar(50)` | 收货人姓名 |
| `consignee_phone` | `varchar(30)` | 联系电话 |
| `consignee_address` | `varchar(255)` | 收货地址 |
| `created_at` | `timestamp` | 下单时间 |
| `updated_at` | `timestamp` | 更新时间 |

### 说明

- 订单主表存放订单级信息
- 一个订单理论上可以包含多个商品，因此商品信息应放在订单明细表
- 即使第一版只做单商品下单，也建议保留标准结构，避免后续重构

## 3.5 订单明细表 `order_items`

用于保存订单中的商品快照。

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `bigserial` | 主键 |
| `order_id` | `bigint` | 关联 `orders.id` |
| `product_id` | `bigint` | 关联 `products.id` |
| `product_name_snapshot` | `varchar(120)` | 下单时商品名称快照 |
| `product_image_snapshot` | `varchar(500)` | 下单时商品图片快照 |
| `unit_price` | `numeric(10,2)` | 下单时单价 |
| `quantity` | `int` | 购买数量 |
| `line_amount` | `numeric(10,2)` | 小计金额 |
| `created_at` | `timestamp` | 创建时间 |

### 说明

- 快照字段非常重要
- 即使商品后续被修改或删除，历史订单仍应保持原始下单信息

## 4. 表关系说明

各表关系如下：

- 一个 `users` 记录对应一个平台用户
- 一个商家用户可以对应一个 `merchant_profiles`
- 一个商家店铺可以拥有多个 `products`
- 一个消费者可以创建多个 `orders`
- 一个 `orders` 可以包含多个 `order_items`

可理解为：

- 用户是账号主体
- 商家店铺是商家业务主体
- 商品归属于商家店铺
- 订单归属于消费者和商家
- 订单明细记录具体买了什么

## 5. 约束建议

## 5.1 唯一约束

- `users.username` 唯一
- `merchant_profiles.user_id` 唯一
- `orders.order_no` 唯一

## 5.2 外键约束

- `merchant_profiles.user_id -> users.id`
- `products.merchant_id -> merchant_profiles.id`
- `orders.consumer_id -> users.id`
- `orders.merchant_id -> merchant_profiles.id`
- `order_items.order_id -> orders.id`
- `order_items.product_id -> products.id`

## 5.3 默认值建议

- `users.status` 默认 `active`
- `merchant_profiles.status` 默认 `approved`
- `products.status` 默认 `active`
- `orders.status` 默认 `pending`
- 所有 `created_at` 默认当前时间
- 所有 `updated_at` 默认当前时间

## 6. 索引建议

为了提升后续查询效率，建议增加以下索引：

- `users(username)`
- `products(merchant_id)`
- `products(status)`
- `orders(consumer_id)`
- `orders(merchant_id)`
- `orders(status)`
- `order_items(order_id)`

## 7. 状态枚举建议

为避免后续状态值混乱，建议状态字段统一采用固定枚举。

### 用户状态

- `active`
- `disabled`

### 商家审核状态

- `pending`
- `approved`
- `rejected`

### 商品状态

- `draft`
- `active`
- `inactive`

### 订单状态

- `pending`
- `accepted`
- `completed`
- `cancelled`

## 8. SQL 建表示例

以下是基于 PostgreSQL 的建表参考示例：

```sql
create table users (
  id bigserial primary key,
  username varchar(50) not null unique,
  password_hash varchar(255) not null,
  role varchar(20) not null check (role in ('merchant', 'consumer')),
  status varchar(20) not null default 'active' check (status in ('active', 'disabled')),
  created_at timestamp not null default now(),
  updated_at timestamp not null default now()
);

create table merchant_profiles (
  id bigserial primary key,
  user_id bigint not null unique references users(id),
  shop_name varchar(100) not null,
  contact_name varchar(50) not null,
  contact_phone varchar(30) not null,
  address varchar(255) not null,
  description text,
  cover_image_url varchar(500),
  status varchar(20) not null default 'approved' check (status in ('pending', 'approved', 'rejected')),
  created_at timestamp not null default now(),
  updated_at timestamp not null default now()
);

create table products (
  id bigserial primary key,
  merchant_id bigint not null references merchant_profiles(id),
  name varchar(120) not null,
  image_url varchar(500),
  description text,
  price numeric(10, 2) not null check (price > 0),
  stock int not null default 0 check (stock >= 0),
  status varchar(20) not null default 'active' check (status in ('draft', 'active', 'inactive')),
  created_at timestamp not null default now(),
  updated_at timestamp not null default now()
);

create table orders (
  id bigserial primary key,
  order_no varchar(40) not null unique,
  consumer_id bigint not null references users(id),
  merchant_id bigint not null references merchant_profiles(id),
  total_amount numeric(10, 2) not null check (total_amount >= 0),
  status varchar(20) not null default 'pending' check (status in ('pending', 'accepted', 'completed', 'cancelled')),
  consignee_name varchar(50) not null,
  consignee_phone varchar(30) not null,
  consignee_address varchar(255) not null,
  created_at timestamp not null default now(),
  updated_at timestamp not null default now()
);

create table order_items (
  id bigserial primary key,
  order_id bigint not null references orders(id),
  product_id bigint not null references products(id),
  product_name_snapshot varchar(120) not null,
  product_image_snapshot varchar(500),
  unit_price numeric(10, 2) not null check (unit_price >= 0),
  quantity int not null check (quantity > 0),
  line_amount numeric(10, 2) not null check (line_amount >= 0),
  created_at timestamp not null default now()
);

create index idx_products_merchant_id on products(merchant_id);
create index idx_products_status on products(status);
create index idx_orders_consumer_id on orders(consumer_id);
create index idx_orders_merchant_id on orders(merchant_id);
create index idx_orders_status on orders(status);
create index idx_order_items_order_id on order_items(order_id);
```

## 9. 后续扩展预留

当前表结构已经为以下扩展留出空间：

- 商家人工审核
- 商品草稿箱
- 一个订单多个商品
- 后续接支付表
- 后续接评价表
- 后续接地址簿

如果后面进入第二阶段，建议再补充这些表：

- `payments`
- `user_addresses`
- `reviews`
- `admin_users`
- `merchant_audit_logs`

## 10. 当前结论

当前 MVP 阶段最核心的数据库设计应围绕：

- 用户
- 商家店铺
- 商品
- 订单
- 订单明细

这 5 张表已经足够支撑第一阶段业务闭环，同时不会把结构做得过重。
