# 类美团 App 原型说明文档

## 1. 文档目的

本文档用于说明当前 `stitch` 目录下原型页面的定位、覆盖范围，以及这些原型与当前 MVP 开发范围之间的关系。

需要特别明确的一点是：

- 原型允许先展示更完整的产品体验
- 原型中可以存在未来能力页面、占位字段和假数据
- 实际开发时，只实现已经确认过的 MVP 需求

因此，不能因为某个页面或字段出现在原型里，就默认它属于第一阶段必须开发的功能。

## 2. 当前开发边界

当前已经确认的 MVP 只实现以下功能：

- 假登录注册
- 商家自动通过入驻
- 商家填写店铺资料
- 商家发布商品
- 商家管理商品
- 消费者浏览商品
- 消费者查看商品详情
- 消费者下单
- 消费者查看自己的订单
- 商家查看并处理订单

当前明确不做真实能力接入：

- 不做手机号验证码登录
- 不做邮箱验证
- 不做微信登录
- 不做真实支付
- 不做物流追踪
- 不做骑手配送
- 不做评价系统
- 不做收藏系统
- 不做搜索筛选真实能力
- 不做推荐算法

如果这些内容出现在原型中，统一视为：

- 视觉占位
- 假数据展示
- 后续版本预留

## 6. 原型中的假数据使用原则

为了避免原型和开发边界混淆，当前项目统一采用以下规则：

### 6.1 可以保留的假数据内容

- 银行卡号
- 支付方式
- 物流状态
- 推荐标签
- 收藏状态
- 评论分数
- 搜索词
- 商品分类标签

### 6.2 这些假数据的含义

它们只表示：

- 页面布局需要这些信息占位
- 未来版本可能扩展到这些能力
- 当前版本的视觉表达先完整设计

它们不表示：

- 当前 MVP 已经实现了这些功能
- 数据库当前必须支持这些完整字段
- 第一阶段后端必须返回这些真实数据

## 7. 开发落地时的执行原则

后续实际实现时，必须遵守以下原则：

### 7.1 以 MVP 文档为准，不以原型中的所有元素为准

开发范围优先遵循：

- [MVP需求文档.md](/Users/hlstart/Ysport/PROJECT/ymt/MVP需求文档.md)
- [技术方案文档.md](/Users/hlstart/Ysport/PROJECT/ymt/技术方案文档.md)
- [表结构设计.md](/Users/hlstart/Ysport/PROJECT/ymt/表结构设计.md)

### 7.2 原型中超出 MVP 的内容处理方式

如果某个原型元素不在 MVP 范围内，处理方式如下：

- 能静态展示的，可保留 UI
- 不能闭环的，可做假数据展示
- 会明显增加开发复杂度的，不实现真实逻辑

### 7.3 当前开发必须坚持的范围

- 只做假登录，不做真实认证接入
- 只做平台内下单，不做真实支付
- 商家注册后自动通过，不做人工审核流
- 只做基本商品模型，不做复杂规格和加料体系
- 只做基础订单状态，不做物流轨迹与配送系统

## 8. 当前结论

当前 `stitch` 原型已经可以作为本项目的 UI 设计基础继续使用。

结论如下：

- 原型允许比 MVP 更完整
- 原型中的未来能力可以先用假数据占位
- 当前开发只实现我们已经确认过的 MVP 功能
- 现有原型已基本覆盖当前 MVP 所需页面

后续开发时，应采用“视觉尽量贴近原型，功能严格收敛到 MVP”的策略推进。
