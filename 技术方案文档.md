# 类美团 App MVP技术方案文档

## 1. 文档目标

本文档用于说明当前 MVP 的技术实现思路，帮助后续快速进入开发阶段，并保证前后端与数据结构设计保持一致。

由于当前项目目录还是空目录，本文档先给出一套适合 MVP 快速落地的推荐方案。目标是：

- 开发简单
- 跑通快
- 结构清晰
- 后续容易扩展

## 2. 总体技术原则

当前 MVP 的技术原则如下：

- 优先实现业务闭环，不追求大而全
- 前后端尽量统一技术栈，降低开发复杂度
- 先保证核心数据模型稳定，再逐步扩展功能
- 先用简单权限模型区分商家与消费者
- 先用最小认证方案，不做复杂账号安全体系

## 3. 推荐技术选型

## 3.1 前端

推荐：

- `Next.js`
- `React`
- `TypeScript`
- `Tailwind CSS`

推荐原因：

- 适合快速搭页面
- 前后端接口可以放在同一项目中，适合 MVP
- 路由清晰，后续扩展方便
- React 生态成熟，适合做商家端和消费者端双界面

## 3.2 后端

推荐：

- 使用 `Next.js Route Handlers` 作为 MVP 后端接口层

推荐原因：

- 当前功能规模不大
- 用户、商品、订单接口都比较标准
- 可以避免一开始拆成两个仓库
- 开发效率更高

后续如果业务复杂度明显提升，再考虑拆出独立后端服务。

## 3.3 数据库

推荐：

- `PostgreSQL`

推荐原因：

- 结构化数据非常适合用户、商家、商品、订单模型
- 事务能力较强，适合处理订单和库存更新
- 后续扩展审核、支付、评价也比较稳

如果只是本地原型，也可以先用 SQLite，但从中短期来看，PostgreSQL 更合适。

## 3.4 ORM

推荐：

- `Prisma`

推荐原因：

- 数据模型定义清晰
- 迁移方便
- 对 Next.js 搭配成熟
- 开发效率高，适合 MVP

## 3.5 鉴权与登录

当前阶段推荐：

- 基于数据库用户表的简单登录
- 登录后生成 session 或 token
- 角色字段区分商家与消费者

由于当前已明确为“假登录”，建议第一阶段采用最轻量方案：

- 注册时写入用户记录
- 登录时校验用户名和密码
- 登录成功后写入 cookie session

不建议当前阶段接入复杂第三方认证服务。

## 3.6 图片存储

当前阶段建议分两步：

第一阶段最小实现：

- 商品图片先支持输入图片 URL

第二阶段再扩展：

- 文件上传到对象存储

这样可以先避免上传系统拖慢开发节奏。

## 4. 系统架构建议

推荐采用“单仓全栈 MVP 架构”：

- 一个 Next.js 项目
- 前端页面与接口统一放在同一仓库
- 使用 Prisma 连接 PostgreSQL

整体结构可以理解为三层：

### 4.1 表现层

负责页面展示与用户交互：

- 登录 / 注册页面
- 消费者商品浏览与下单页面
- 商家工作台页面

### 4.2 接口层

负责处理请求和返回结果：

- 用户注册 / 登录接口
- 商家店铺信息接口
- 商品管理接口
- 订单管理接口

### 4.3 数据层

负责业务数据持久化：

- 用户表
- 商家表
- 商品表
- 订单表
- 订单明细表

## 5. 模块划分建议

## 5.1 认证模块

职责：

- 注册
- 登录
- 退出登录
- 获取当前登录用户

最小接口建议：

- `POST /api/auth/register`
- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`

## 5.2 商家模块

职责：

- 创建 / 更新店铺资料
- 获取当前商家店铺信息

最小接口建议：

- `GET /api/merchant/profile`
- `POST /api/merchant/profile`
- `PUT /api/merchant/profile`

## 5.3 商品模块

职责：

- 商家新增商品
- 商家编辑商品
- 商家上下架商品
- 消费者查看商品列表和详情

最小接口建议：

- `GET /api/products`
- `GET /api/products/:id`
- `POST /api/products`
- `PUT /api/products/:id`
- `PATCH /api/products/:id/status`

## 5.4 订单模块

职责：

- 消费者创建订单
- 消费者查看自己的订单
- 商家查看自己收到的订单
- 商家更新订单状态

最小接口建议：

- `POST /api/orders`
- `GET /api/orders/my`
- `GET /api/orders/merchant`
- `PATCH /api/orders/:id/status`

## 6. 页面路由建议

推荐路由结构如下：

### 公共页面

- `/login`
- `/register`

### 消费者端页面

- `/`
- `/products/[id]`
- `/checkout/[id]`
- `/orders`

### 商家端页面

- `/merchant`
- `/merchant/profile`
- `/merchant/products`
- `/merchant/products/new`
- `/merchant/products/[id]/edit`
- `/merchant/orders`

## 7. 权限设计

当前 MVP 采用最小权限模型，只区分两种角色：

- `merchant`
- `consumer`

### 权限规则

- `merchant` 可以访问商家后台相关页面与接口
- `consumer` 可以访问前台浏览、下单与个人订单页面
- 未登录用户只能访问公开页面和登录注册页面

### 需要保护的资源

- 商家商品管理接口
- 商家订单管理接口
- 消费者个人订单接口
- 当前用户资料接口

## 8. 关键业务规则

## 8.1 注册与登录

- 用户注册时必须选择角色
- 用户名需要唯一
- 登录必须校验用户名和密码

## 8.2 商家资料

- 商家登录后应尽快补全店铺资料
- 商家状态虽然当前默认自动通过，但数据库中必须保留状态字段

## 8.3 商品发布

- 商品必须归属某个商家
- 商品价格必须大于 0
- 商品库存不能小于 0
- 只有上架商品可以展示给消费者

## 8.4 下单

- 下单时必须校验商品是否存在且已上架
- 下单时必须校验库存是否足够
- 订单创建成功后需要扣减库存
- 订单中必须记录商品快照信息，至少包含下单单价和商品名称

## 9. 前端实现建议

为了适合 MVP 快速开发，前端建议遵循以下原则：

- 页面尽量简洁，不做复杂动画
- 先保证信息结构清晰，再优化视觉
- 商家端和消费者端可共用基础组件
- 表单统一使用同一种表单方案

建议页面结构：

- 消费者首页优先展示商品卡片列表
- 商品详情页展示商品信息和下单入口
- 商家后台优先突出“商品管理”和“订单管理”

## 10. 后端实现建议

后端建议采用“路由处理器 + 服务层”的简单分层：

- Route Handler 负责参数接收和响应格式
- Service 层负责业务逻辑
- Prisma 负责数据库访问

这样做的好处是：

- 代码不会全部堆在接口文件里
- 后续扩支付、审核、优惠时更容易维护

## 12. 开发阶段建议

建议按下面顺序开发：

### 第一阶段

- 初始化项目
- 建立数据库
- 完成用户注册登录

### 第二阶段

- 完成商家资料管理
- 完成商品发布和展示

### 第三阶段

- 完成下单流程
- 完成订单列表与状态流转

### 第四阶段

- 完成基础联调
- 完成最小验收测试

## 13. 风险与注意事项

当前阶段需要特别注意以下问题：

- 假登录方案仅适用于 MVP 验证，不适合正式上线
- 如果一开始不保存订单快照，后面会导致历史订单数据失真
- 如果订单和库存更新不做事务处理，后续容易出现超卖问题
- 如果页面和接口不提前做角色隔离，后面补权限会比较痛苦

## 14. 当前结论

当前项目最适合采用：

- 单仓 Next.js 全栈方案
- PostgreSQL + Prisma
- App Router 页面与 API 同仓开发
- 轻量 session 登录

这套方案足够支撑当前 MVP，也为后续人工审核、支付、评价、配送等能力留下了扩展空间。
