# 学习总结写入 Inbox

把聊天中的学习内容、排查结论、调试记录整理成 Markdown 文档，并写入用户的 HeLoNote inbox 文件夹，方便后续复盘和沉淀。

## 适用场景

当用户出现下面这类意思时，使用这个 skill：

- 总结学习
- 总结这次学习
- 整理学习记录
- 生成学习总结
- 把这次内容总结成 md
- 总结到 inbox
- 把这次排查记录下来
- 帮我生成一个笔记 / 文档留档

## 输出位置

默认把 Markdown 文档写到：

```text
/Users/hlstart/Ysport/PROJECT/HeLoNote/inbox
```

除非用户明确要求放到别的位置，否则不要把这类总结文件写回当前项目仓库。

## 工作流程

1. 先明确这次要总结的主题。
2. 从当前对话、代码、报错信息、确认过的结论里提取关键信息。
3. 把“已经确认的事实”和“曾经怀疑但已被排除的假设”分开写清楚。
4. 整理成结构化 Markdown 文档。
5. 将 `.md` 文件保存到 HeLoNote inbox 文件夹。
6. 回复用户最终生成的绝对路径。

## 默认文档结构

如果用户没有指定格式，优先使用下面这些标题，并按实际情况增删：

- `# 标题`
- `## 背景 / 问题概述`
- `## 现象`
- `## 错误信息 / 错误代码`
- `## 根因分析`
- `## 解决步骤`
- `## 最终结论`
- `## 后续建议`

如果这是学习总结而不是纯 bug 排查，也可以改成更贴合内容的结构，例如：

- `## 学到了什么`
- `## 关键概念`
- `## 走过的弯路`
- `## 正确做法`
- `## 经验总结`

## 写作规则

- 使用 Markdown。
- 默认使用中文；除非用户明确要求英文。
- 内容尽量简洁，但要完整可复盘。
- 错误码、报错原文、返回值尽量用代码块标出。
- 区分清楚：是代码问题、环境变量问题、部署问题，还是浏览器环境问题。
- 不要把未证实的猜测写成既定事实。
- 如果某个怀疑后来被排除了，也要明确写出来。

## 文件命名

如果用户没有指定文件名，就根据主题自动生成一个简洁的 `.md` 文件名。

优先使用英文 kebab-case，便于管理，例如：

- `turnstile-debug-summary.md`
- `nextjs-auth-learning-note.md`
- `postgres-schema-mismatch-summary.md`

如果用户指定中文文件名，也可以直接使用。

## 写入前检查

创建文件前，先做这些检查：

1. 确认目标路径是 HeLoNote inbox。
2. 确认用户是否指定了文件名。
3. 如果目标文件可能已存在，先读取后决定覆盖还是改名。

## 完成后的回复

写完文件后，回复内容应包含：

- 一句简短确认
- 最终文件的绝对路径

## 说明

这个 skill 的目标不是只在聊天里总结，而是产出一个可以长期保存、后续复盘可直接查看的 Markdown 文档。
