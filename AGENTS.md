# AGENTS.md

本文件用于约束 Codex 在当前项目目录及其子目录中的默认执行方式。

## 1. 项目阶段

当前项目只开发已经确认过的 MVP。

默认规则：

- 不主动扩需求
- 不把原型中的全部内容默认视为当前开发范围
- 只实现已经和用户明确确认过的 MVP 功能

## 2. 自主开发权限

在已经沟通过的需求范围内，Codex 可以自主推进开发。

默认允许自主决定的事项：

- 技术实现细节
- 代码结构和模块拆分
- 接口组织方式
- 小范围字段命名调整
- 小范围页面结构调整
- 与原型一致前提下的轻微文案微调
- 文档补充
- 本地可执行的验证与测试

前提是：

- 不超出已确认的 MVP
- 不改变产品方向
- 不明显偏离原型主体

## 3. 必须先确认的情况

以下事项必须先询问用户，再继续：

- 产品方向变化
- 新增非 MVP 功能
- 原型大改
- 删除已有设计
- 出现此前未沟通过但必须新增的需求
- 超出本项目边界的事项

## 4. 原型执行规则

当前项目默认以 `stitch` 原型为主要 UI 参考。

执行规则：

- 视觉尽量贴近原型
- 原型可以比当前 MVP 更完整
- 原型中超出 MVP 的功能只做静态展示、假数据占位或暂不实现逻辑
- 不因为原型里出现某个能力，就自动把它纳入当前开发

## 5. 代码注释要求

用户是非技术背景，代码必须尽量写清楚注释。

默认要求：

- 组件要有用途说明
- 接口要有作用说明
- 方法要尽量有说明
- 关键逻辑、条件判断、状态流转都要写注释
- 对初学者不直观的代码优先补注释

目标：

- 让用户能看懂代码在做什么
- 让后续维护时更容易理解为什么这么写

## 6. 技术决策平衡原则

当前虽然是 MVP，但不能完全忽视后续扩展。

默认原则：

- 优先满足当前 MVP
- 如果某种实现能兼顾后续扩展且不会明显增加当前复杂度，优先采用更平衡的方案
- 避免过度设计
- 避免明显会导致后续大面积返工的短期方案

## 7. 测试与验证

每次实现后，默认先自行完成能做的验证。

要求：

- 能运行就先运行
- 能测试就先测试
- 能检查就先检查
- 无法验证的部分要明确说明原因

## 8. 文档与配置

默认采用“重要事项尽量落文件”的工作方式。

要求：

- 重要决策尽量写进根目录文档
- 不只在对话中口头说明
- 该写进配置的内容要写进配置
- 实现过程中如规则变化，应同步更新相关文档

## 9. 页面与字段调整

在不改变核心需求的前提下，可以做适度工程化调整。

允许：

- 小幅调整页面结构以适配实现
- 小幅调整字段命名以保证代码一致性
- 小幅调整原型中的占位文案

限制：

- 主体不能发生太大变化
- 不得明显偏离原型

## 10. 默认优先级

默认优先级顺序：

1. 不超出已确认 MVP 范围
2. 尽量贴近原型
3. 在边界内尽量自主开发
4. 保持代码清晰、注释完整
5. 在合理范围内兼顾后续扩展

## 11. 默认工作方式

后续默认工作方式如下：

- 先自主分析现有文档、原型和代码
- 在已确认范围内直接推进实现
- 小问题自行处理
- 重大分歧或超范围事项再停下来确认
- 阶段完成后说明已做内容、验证情况和剩余风险
